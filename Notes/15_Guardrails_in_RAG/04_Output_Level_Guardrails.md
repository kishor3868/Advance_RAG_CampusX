# 1. Introduction to Output-Level Guardrails

In the previous sections we discussed

- Query-Level Guardrails
- Retrieval-Level Guardrails

Both of these stages ensure that the input query and retrieved context are safe.

However, there is still one important question remaining:

> **Can we directly trust the answer generated by the LLM?**

The answer is **No**.

Even after receiving a good query and relevant retrieved documents, the LLM can still:

- Hallucinate
- Generate harmful content
- Leak sensitive information
- Produce incorrect JSON
- Ignore the retrieved context

Therefore, before sending the response back to the user, we perform a **second round of validation**, known as **Output-Level Guardrails**.

Output Guardrails act as the **final quality and safety checkpoint** of the RAG pipeline.

---

# 2. Why are Output-Level Guardrails Required?

Consider the following RAG pipeline:

```
User Query

↓

Retriever

↓

Retrieved Context

↓

LLM

↓

Generated Response
```

Even if the retrieved context is completely correct, the LLM may still generate:

- Facts not present in the context
- Toxic language
- Biased statements
- Sensitive information
- Invalid output formats

Example

Retrieved Context

```
Tesla CEO is Elon Musk.
```

Generated Response

```
Tesla CEO is Elon Musk.

Tesla was founded in 1990.
```

The second statement is unsupported.

Therefore, the response should not be sent directly to the user.

Instead,

Output-Level Guardrails verify the response before delivery.

# 3. Faithfulness Check

## Definition

**Faithfulness Check verifies whether every statement in the generated response is supported by the retrieved documents.**

It ensures that the LLM does not invent new information.

Faithfulness is one of the most important Output Guardrails because it directly checks whether the response is grounded in the retrieved knowledge.

---

## Why is it Needed?

LLMs sometimes mix retrieved facts with their own internal knowledge.

Example

Retrieved Context

```
Graph RAG combines Knowledge Graphs with Retrieval-Augmented Generation.
```

Generated Response

```
Graph RAG combines Knowledge Graphs with Retrieval-Augmented Generation.

It was invented by Google in 2022.
```

The second statement is not present in the retrieved context.

Faithfulness Check detects this unsupported claim.

---

## Architecture

```
Retrieved Context

↓

Generated Response

↓

Judge LLM

↓

Every Claim Supported?

YES

↓

Return Response

NO

↓

Reject / Retry
```

---

## Example

Retrieved Context

```
Elon Musk is CEO of Tesla.

Tesla headquarters is Austin.
```

Generated Response

```
Elon Musk is CEO.

Tesla headquarters is Austin.

Tesla manufactures aircraft.
```

Judge LLM verifies

```
Claim 1

Supported

✔

----------------

Claim 2

Supported

✔

----------------

Claim 3

Unsupported

✘
```

Faithfulness Score becomes lower because one claim is unsupported.

---

## LangChain Example

```python
from ragas.metrics import faithfulness
from ragas import evaluate

result = evaluate(
    dataset,
    metrics=[faithfulness]
)

print(result)
```

# 4. Hallucination Detection

## Definition

**Hallucination Detection identifies statements generated by the LLM that are not supported by the retrieved context or known facts.**

Hallucination means the LLM confidently generates information that it cannot justify.

---

## Why is it Needed?

Hallucinations are one of the biggest challenges in production RAG systems.

Example

Question

```
Who invented Graph RAG?
```

Retrieved Context

```
Graph RAG combines Graph Databases with Retrieval-Augmented Generation.
```

Generated Response

```
Graph RAG was invented by Elon Musk.
```

The retrieved documents never mention Elon Musk.

This is a hallucination.

---

## Architecture

```
Retrieved Context

↓

Generated Response

↓

Claim Extraction

↓

Evidence Search

↓

Supported?

YES

↓

Safe Response

NO

↓

Hallucination Detected
```

---

## Example

Retrieved Context

```
Earth revolves around the Sun.
```

Generated Response

```
Earth revolves around the Moon.
```

Since no supporting evidence exists,

Hallucination Guardrail blocks the response.

---

## LangChain Example

```python
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4.1")

hallucination_prompt = """
You are a fact checker.

Retrieved Context:

{context}

Generated Answer:

{answer}

Determine whether every claim is supported.

Reply ONLY

SUPPORTED

or

HALLUCINATED
"""

response = llm.invoke(
    hallucination_prompt.format(
        context=context,
        answer=answer
    )
)

print(response.content)
```

# 5. Toxicity / Bias Filtering

## Definition

**Toxicity/Bias Filtering ensures that the generated response does not contain abusive, hateful, offensive, discriminatory, or biased language before it is delivered to the user.**

---

## Why is it Needed?

Even if the retrieved context is correct,

the LLM may produce:

- Hate speech
- Offensive language
- Gender bias
- Religious bias
- Racial bias

These responses should never reach the end user.

---

## Architecture

```
Generated Response

↓

Safety Classifier

↓

Safe?

YES

↓

Return Response

NO

↓

Reject Response
```

---

## Example

Generated Response

```
People from a particular community are inferior.
```

Safety Classifier

↓

Unsafe

↓

Blocked

---

## LangChain Example

```python
toxicity_prompt = """
Check whether the response contains

- Hate Speech
- Toxicity
- Bias

Return

SAFE

or

UNSAFE

Response:

{text}
"""

result = llm.invoke(
    toxicity_prompt.format(
        text=response
    )
)

print(result.content)
```

# 6. PII Detection

## Definition

**Output-Level PII Detection identifies sensitive personal information present in the generated response before it is shown to the user.**

Unlike Retrieval-Level PII Scrubbing,

here we inspect the **final generated answer**.

---

## Why is it Needed?

Suppose the Retriever contains employee records.

User asks

```
Tell me about employee policies.
```

Generated Response

```
Employee Salary

₹25,00,000

PAN

ABCDE1234F

Phone

9876543210
```

This response leaks confidential information.

Output Guardrails detect and remove it.

---

## Architecture

```
Generated Response

↓

PII Detector

↓

Sensitive Data?

YES

↓

Mask

↓

Return Response
```

---

## Example

Before

```
Phone Number

9876543210
```

After

```
Phone Number

**********
```

---

## LangChain Example

```python
pii_prompt = """
Detect whether the response contains

- Phone Numbers
- PAN
- Aadhaar
- Email
- Credit Card

Return

SAFE

or

PII FOUND

Text:

{text}
"""

result = llm.invoke(
    pii_prompt.format(
        text=response
    )
)

print(result.content)
```

# 7. Format Validation

## Definition

**Format Validation ensures that the generated response follows the expected output structure before it is returned to the application.**

This is especially important when downstream applications expect structured data.

---

## Why is it Needed?

Suppose your application expects JSON.

Expected Output

```json
{
  "name":"Graph RAG",
  "type":"Retrieval Technique"
}
```

Generated Output

```
Graph RAG is a retrieval method.

It is useful.
```

Although the answer is correct,

the application will fail because it expects JSON.

---

## Architecture

```
Generated Response

↓

Schema Validator

↓

Valid Format?

YES

↓

Return Response

NO

↓

Regenerate Response
```

---

## Example

Expected

```json
{
  "city":"Delhi",
  "temperature":35
}
```

Generated

```
Delhi is hot today.
```

Format Validation fails.

---

## LangChain Example

Using Pydantic Output Parser

```python
from pydantic import BaseModel

class Answer(BaseModel):
    city: str
    temperature: int
```

```python
from langchain.output_parsers import PydanticOutputParser

parser = PydanticOutputParser(
    pydantic_object=Answer
)

format_instructions = parser.get_format_instructions()
```

The parser validates that the generated response matches the required schema.

# 9. Combined LangChain Workflow

```python
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4.1")

response = llm.invoke(user_query)

# Faithfulness Check
# Hallucination Detection
# Toxicity Detection
# PII Detection
# Format Validation

# If every guardrail passes

print(response.content)
```

In production, these validations are typically chained together. If any guardrail fails, the system may:
- Reject the response,
- Regenerate the answer,
- Ask the retriever for better context,
- Or escalate the request for human review.

# 10. Key Takeaways

- **Output-Level Guardrails** act as the **final safety and quality checkpoint** before an LLM response is delivered to the user.
- **Faithfulness Check** verifies that every statement in the generated response is supported by the retrieved documents, ensuring the answer remains grounded.
- **Hallucination Detection** identifies unsupported or fabricated claims generated by the LLM, helping prevent misinformation.
- **Toxicity/Bias Filtering** blocks responses containing harmful, offensive, or discriminatory language, ensuring safe and responsible AI behavior.
- **Output-Level PII Detection** prevents accidental disclosure of sensitive personal or confidential information contained in the generated response.
- **Format Validation** ensures that the response follows the expected schema or structure (such as JSON), making it suitable for downstream applications and APIs.
- Together, these output guardrails ensure that the final response is **factually grounded, safe, privacy-preserving, properly formatted, and ready for production use**.
