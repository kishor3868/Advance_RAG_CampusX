# 1. Introduction

In **Strategy 1**, we solved the Representation Problem by converting images into text using a **Vision Language Model (VLM)**.

Although this approach is simple, it has one major drawback:

> The image is no longer represented in its original visual form. Instead, it is represented by the textual description generated by the VLM.

If the VLM misses some visual details or hallucinates, retrieval quality decreases.

To overcome this limitation, we use **Strategy 2**, where both text and images are directly converted into embeddings without first converting images into text.

---

# 2. Why do we need Strategy 2?

Suppose we have an image containing:

- A complex medical scan
- A scientific diagram
- An engineering blueprint

If we convert these images into text, some visual details may be lost.

Instead, it is better to directly generate an embedding from the image itself.

This leads to the use of **Multimodal Embedding Models**.

---

# 3. What are Multimodal Embedding Models?

## Definition

A **Multimodal Embedding Model** is an embedding model capable of generating embeddings for different modalities (such as text and images) in a **shared embedding space**, so that semantically similar text and images are represented by nearby vectors.

Unlike traditional embedding models, these models understand both:

- Text
- Images

Examples:

- CLIP (OpenAI)
- SigLIP
- BLIP
- ImageBind
- Voyage Multimodal Embeddings
- Gemini Embeddings (Multimodal)

# 4. Why Separate Text and Image Embeddings Don't Work?

Suppose we use:

```
Text

↓

Text Embedding Model

↓

Vector A
```

and

```
Image

↓

Image Embedding Model

↓

Vector B
```

The problem is:

Both models are trained independently.

Therefore,

- Different dimensions
- Different feature spaces
- Different semantic meanings

As a result,

we cannot directly compare:

```
Text Vector

with

Image Vector
```

This makes similarity search impossible.

---

## Example

Query

```
Tesla Factory
```

Text Embedding

```
[0.21,0.35,0.61]
```

Image Embedding

```
[0.92,0.14,0.48]
```

These vectors are generated by different models.

Their values do not represent the same semantic space.

Therefore,

Cosine Similarity is meaningless.

---

# 5. Shared Embedding Space

The solution is to use a **Multimodal Embedding Model** that maps both text and images into the **same vector space**.

---

## Shared Embedding Space

```
                 Shared Embedding Space

        Car Image

             ●

                    Tesla Text

                       ●

                               Factory Image

                                  ●

                     Cat Image

●

Dog Image

       ●
```

Notice:

Semantically related text and images are placed close together.

This allows us to compare:

- Text ↔ Text
- Image ↔ Image
- Text ↔ Image

using the same similarity metric.

---

# Why Shared Embedding Space?

Suppose user asks:

```
Show me Tesla factory images.
```

Query

↓

Embedding

↓

Similarity Search

↓

Image Embeddings

Because both text and images share the same embedding space,

the text query can retrieve relevant images directly.

This completely solves the **Retrieval Problem**.

---

# 6. Dimensionality Alignment

Another important requirement is:

Both text and image embeddings must have the **same dimensionality**.

Example

```
Text Embedding

768 dimensions
```

```
Image Embedding

768 dimensions
```

Now,

Cosine Similarity can be calculated correctly.

---

Without dimensionality alignment:

```
Text

384 dimensions

Image

1024 dimensions
```

Similarity cannot be computed.

Therefore,

Multimodal Embedding Models ensure:

- Same dimensions
- Same feature representation
- Same semantic space

# 10. LangChain Implementation

## Step 1 : Load Documents

```python
from langchain_community.document_loaders import PyPDFLoader

loader = PyPDFLoader(
    "report.pdf",
    extract_images=True
)

documents = loader.load()
```

---

Pseudo Workflow

```python
Text

↓
```


(Depending on the provider, a multimodal embedding model such as CLIP, Voyage AI, or Gemini Embeddings would be used.)

---

## Step 3 : Store in Vector Database

```python
from langchain_chroma import Chroma
```

```python

results = retriever.invoke(
```

# 11. Strategy 1 vs Strategy 2

| Strategy 1 (VLM Description) | Strategy 2 (Multimodal Embeddings) |
|------------------------------|------------------------------------|
| Images converted to text | Images embedded directly |
| Uses VLM during ingestion | Uses Multimodal Embedding Model |
| Simpler pipeline | More advanced pipeline |
| May lose visual details | Preserves visual features |
| Depends heavily on prompt quality | Depends on embedding model quality |
| Lower accuracy for complex visuals | Better accuracy for visual retrieval |
| Easier to implement | More computationally intensive |

---

# 12. Advantages

- Better visual understanding.
- Preserves image features.
- Supports Text ↔ Image retrieval.
- Higher retrieval accuracy.
- No information loss caused by image descriptions.
- Better for complex diagrams, charts, and medical images.

---

# 13. Disadvantages

- Requires specialized multimodal embedding models.
- More expensive than Strategy 1.
- Higher computational cost.
- Limited availability of high-quality multimodal embedding models.
- Larger storage requirements for image embeddings.

---

# 14. Interview Questions

### Q1. Why do we need Strategy 2?

To avoid information loss caused by converting images into text and to enable direct retrieval of images using embeddings.

---

### Q2. What is a Multimodal Embedding Model?

A model that generates embeddings for both text and images in a shared embedding space.

---

### Q3. Why can't separate text and image embeddings be compared?

Because they are generated by different models with different feature spaces and possibly different dimensions.

---

### Q4. What is a Shared Embedding Space?

A common vector space where semantically similar text and images are placed close together.

---

### Q5. Does Strategy 2 eliminate the need for a Vision Language Model?

No.

The embedding model is responsible for **retrieval**, while the Vision Language Model is still required for **reasoning and answer generation**.

---

# 15. Key Takeaways

- **Strategy 2** solves the Representation and Retrieval Problems using **Multimodal Embedding Models** instead of converting images into text.
- Both **text** and **images** are embedded into a **shared embedding space**, allowing direct comparison using similarity search.
- The embedding model ensures **dimensionality alignment**, so text and image vectors have compatible dimensions and semantic meaning.
- During retrieval, a text query can directly retrieve both **text documents** and **relevant images** from the vector database.
- A **Vision Language Model (VLM)** is still used after retrieval to combine the user query, retrieved text, and retrieved images into a coherent final response.
- Compared to Strategy 1, this approach offers **higher retrieval accuracy** and preserves visual information better, but requires more specialized models and higher computational resources.
